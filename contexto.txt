Crie um sistema web completo, funcional e responsivo para gestão operacional de Rally, com foco principal em mobile first, utilizando React no frontend e Node.js com Express no backend.

Objetivo principal:
Construir uma base sólida, escalável, organizada e pronta para evolução futura, considerando que todo o sistema gira em torno do contexto de ano + edição do Rally.

Arquitetura obrigatória do projeto:
- O projeto deve possuir obrigatoriamente duas pastas na raiz:
  - /frontend
  - /backend
- Todo o código React deve ficar dentro de /frontend
- Todo o código da API e servidor deve ficar dentro de /backend
- Não misturar frontend e backend
- Estruturar o projeto para fácil manutenção, escalabilidade e clareza
- Reaproveitar ao máximo componentes, hooks, funções, serviços e utilitários
- Não duplicar código
- Antes de criar novos componentes, verificar se já existe alguma base reutilizável que possa ser adaptada
- Separar claramente UI, regras de negócio, serviços, tipos, contextos, utilitários e integrações

Stack obrigatória do frontend:
- React
- Vite
- TypeScript

- React Router
- mobile first
- componentização forte

Stack obrigatória do backend:
- Node.js

- organização por módulos/domínios
- estrutura preparada para futura integração com banco de dados

Contexto principal do negócio:
O Rally é organizado por ano e edição.
Cada edição possui sua própria operação.
Todos os dados operacionais devem estar vinculados à edição selecionada.

Fluxo obrigatório do sistema:
- Ao acessar o sistema, o usuário deve primeiro escolher o ano do Rally
- Depois deve escolher a edição daquele ano
- Somente após selecionar ano e edição o usuário poderá visualizar e gerenciar os dados relacionados àquela edição
- Nenhum dado operacional pode aparecer antes da seleção de ano e edição
- Todas as telas e consultas devem respeitar o contexto da edição selecionada

Domínio do sistema:

Regras de negócio obrigatórias:
- Todo dado operacional pertence a uma edição específica
- O usuário nunca deve visualizar dados soltos sem antes escolher ano e edição
- Cada equipe pertence a uma edição
- Cada carro pertence a uma edição
- Cada rota pertence a uma edição
- Cada hotel pode ser vinculado à edição
- Cada equipe pode ter várias pessoas
- No cadastro de pessoa deve existir um checkbox com o texto “É responsável pela equipe?”
- O sistema deve permitir apenas um responsável principal por equipe
- Quando uma pessoa estiver marcada como responsável, isso deve aparecer com destaque visual na interface
- Cada equipe deve poder ter rota vinculada
- Cada hotel pode ser editado dentro do contexto operacional
- Deve existir controle de quilometragem por carro
- Deve existir controle de combustível por carro
- Deve existir controle de manutenção por carro
- O carro deve exibir seu histórico completo dentro da edição selecionada
- O controle de alimentação deve funcionar por pessoa e por total da equipe
- Deve ser possível cadastrar um limite de alimentação por equipe
- O limite é apenas informativo
- O sistema deve avisar quando o consumo estiver próximo do limite
- O sistema deve avisar quando o consumo ultrapassar o limite
- O sistema não deve bloquear lançamentos quando ultrapassar o valor

Regra de alimentação:
- calcular total por pessoa
- calcular total consolidado por equipe
- exibir limite configurado
- exibir valor consumido
- criar status visual para consumo:
  - normal
  - warning
  - exceeded

Criar utilitário obrigatório reutilizável:
- função que recebe valorAtual e limite
- retorna:
  - normal
  - warning
  - exceeded
- usar esse utilitário em todo o sistema onde houver controle de limite

Paleta e design system obrigatórios:
Criar um design system centralizado utilizando Tailwind CSS.
Não usar cores hardcoded diretamente dentro dos componentes.

Configurar o Tailwind com as seguintes cores:
theme: {
  extend: {
    colors: {
      background: '#F8EDD9',
      primary: '#317039',
      surface: '#FFFFFF',
      accent: '#F1BE49',
      dark: '#143022',
      text: '#1A1A1A',
    }
  }
}

Regras visuais obrigatórias:
- usar background como fundo principal
- usar surface para cards e containers
- usar primary para ações principais
- usar accent para destaques e avisos
- usar dark para títulos, detalhes e pontos de contraste
- usar text como cor principal de texto
- nunca usar #hex diretamente nos componentes
